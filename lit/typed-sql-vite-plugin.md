# Type-Safe SQL via YAML Front Matter + Vite Plugin

## Problem

Four layers of manual typing that can drift out of sync:

1. **Migration SQL** (`migrations/001_create_tables.sql`) — source of truth for schema
2. **Model interfaces** (`src/data/*/model.ts`) — hand-written to match migrations
3. **SQL query files** (`src/data/tenet/queries/*.sql`) — no autocomplete for tables/columns, no param validation
4. **Repository methods** — result types specified via generics (`queryOne<TenetRow>`), param objects must manually match `@paramName` placeholders

Any schema change requires coordinated updates across all four layers with zero compiler help.

## Goal

Keep raw SQL in `.sql` files. Get:

- **Table/column types** auto-generated from migrations
- **Typed query params and results** via YAML front matter annotations in SQL files
- **Zero drift** between migrations, models, and queries
- **All through a Vite plugin** — no custom scripts, no ORMs

---

## 1. YAML Front Matter Spec

Each `.sql` file gets an optional YAML front matter block delimited by `---` on its own lines. Two fields: `params` (a map of param name to TypeScript type) and `result` (a TypeScript type expression).

### Simple SELECT with params

```sql
---
params:
  slug: string
result: tenets
---
SELECT * FROM tenets WHERE slug = @slug
```

### SELECT with a JOIN (extended result type)

```sql
---
result: "tenets & { proposer_login: string, proposer_avatar: string | null }"
---
SELECT t.*, u.login AS proposer_login, u.avatar_url AS proposer_avatar
FROM tenets t
JOIN users u ON u.id = t.proposed_by_id
ORDER BY t.created_at DESC
```

### INSERT with nullable params

```sql
---
params:
  tenetId: number
  title: string
  description: "string | null"
  pros: "string | null"
  cons: "string | null"
  sortOrder: number
result: void
---
INSERT INTO tenet_options (tenet_id, title, description, pros, cons, sort_order)
VALUES (@tenetId, @title, @description, @pros, @cons, @sortOrder)
```

### UPDATE

```sql
---
params:
  id: number
  status: TenetStatus
result: void
---
UPDATE tenets SET status = @status, updated_at = datetime('now') WHERE id = @id
```

### No params, no front matter

```sql
-- getOptions.sql (front matter optional)
---
params:
  tenetId: number
result: tenet_options
---
SELECT * FROM tenet_options WHERE tenet_id = @tenetId ORDER BY sort_order
```

### Rules

| Field | Required | Values |
|---|---|---|
| `params` | No | Map of `name: type`. Omit for parameterless queries |
| `result` | No | Table name (`tenets`), TS type expression (`"tenets & { extra: string }"`), or `void` for mutations. Defaults to `Record<string, unknown>` if omitted |

The `result` field references types from the generated `db-types.d.ts` by default (bare table names), or any TypeScript type expression when quoted.

## 2. Schema Codegen: Migrations to `db-types.d.ts`

The plugin parses `CREATE TABLE` statements from migration files and generates TypeScript interfaces.

### Input: `migrations/001_create_tables.sql`

```sql
CREATE TABLE IF NOT EXISTS tenets (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    title           TEXT NOT NULL,
    slug            TEXT NOT NULL UNIQUE,
    status          TEXT NOT NULL DEFAULT 'draft'
                    CHECK(status IN ('draft','voting','accepted','rejected','implemented','superseded')),
    context         TEXT NOT NULL,
    decision        TEXT,
    rationale       TEXT,
    proposed_by_id  INTEGER NOT NULL REFERENCES users(id),
    created_at      TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at      TEXT NOT NULL DEFAULT (datetime('now')),
    superseded_by_id INTEGER REFERENCES tenets(id)
);
```

### Output: `src/data/db-types.d.ts`


### Type Mapping Rules

| SQLite type | TypeScript type |
|---|---|
| `INTEGER` | `number` |
| `TEXT` | `string` |
| `REAL` | `number` |
| `BLOB` | `ArrayBuffer` |
| Column without `NOT NULL` | `T \| null` |
| `CHECK(col IN ('a','b','c'))` | `"a" \| "b" \| "c"` (union literal) |
| `INTEGER PRIMARY KEY` | `number` (never null) |

The CHECK constraint extraction means generated `status` and `choice` columns are already typed as string unions — no need for hand-written `TenetStatus` or `VoteChoice` types (though they can be kept as aliases).

### Table Name Overrides

By default, the plugin converts table names to PascalCase singular form (e.g., `tenets` → `Tenet`, `user_profiles` → `UserProfile`). For irregular plurals or custom naming preferences, you can override this behavior:

```ts
// vite.config.ts
import { sqlTypesPlugin } from "./.vite/plugins";

export default defineConfig({
  plugins: [
    sqlTypesPlugin({
      tableNameOverrides: {
        "people": "Person",
        "children": "Child",
        "data": "Datum",
        "user_profiles": "UserProfile", // override automatic conversion
      },
    }),
  ],
});
```

With this configuration, a table named `people` will generate `export interface Person` instead of the default `People`.

## 3. Vite Plugin Architecture

The plugin has three responsibilities:

### `buildStart` — Generate types from migrations + SQL files

```
migrations/*.sql  ──parse──>  src/data/db-types.d.ts
src/**/*.sql      ──parse──>  src/**/queries.generated.ts  (per-directory)
```

### `transform` — Strip front matter from SQL at build time

When Vite encounters a `.sql` import, the plugin strips the YAML front matter and returns the clean SQL string. This replaces the need for `?raw`.

```ts
// Input: findBySlug.sql (with front matter)
// Output: "SELECT * FROM tenets WHERE slug = ?" (clean SQL)
```

### `configureServer` — Watch and regenerate in dev

Watch `migrations/` and `**/*.sql` files. On change, re-parse and regenerate the affected `.d.ts` files. Trigger HMR for the consuming modules.

### Plugin Skeleton

```ts
// .vite/plugins/sql-types.ts
import type { Plugin } from "vite";
import matter from "gray-matter";
import { parseMigrations } from "./sql-types/parse-migrations";
import { generateDbTypes } from "./sql-types/generate-db-types";
import { generateQueryBarrel } from "./sql-types/generate-query-barrel";

export interface SqlTypesPluginOptions {
  /**
   * Override automatic table name to type name conversions.
   * Map of table names (as they appear in the database) to desired TypeScript type names.
   * 
   * @example
   * {
   *   "people": "Person",
   *   "children": "Child",
   *   "user_profiles": "UserProfile"
   * }
   */
  tableNameOverrides?: Record<string, string>;
}

export function sqlTypesPlugin(options: SqlTypesPluginOptions = {}): Plugin {
  const { tableNameOverrides = {} } = options;
  
  return {
    name: "sql-types",

    async buildStart() {
      // 1. Parse all migration files -> generate db-types.d.ts
      const dbTypes = await parseMigrations("migrations/");
      await generateDbTypes(dbTypes, "src/data/db-types.d.ts", tableNameOverrides);

      // 2. Scan all .sql query files -> generate typed barrels per directory
      const sqlFiles = await glob("src/data/**/*.sql");
      const byDir = groupByDir(sqlFiles);
      for (const [dir, files] of Object.entries(byDir)) {
        await generateQueryBarrel(dir, files, tableNameOverrides);
      }
    },

    transform(code, id) {
      if (!id.endsWith(".sql")) return;
      const { content } = matter(code);
      return { code: `export default ${JSON.stringify(content.trim())};`, map: null };
    },

    configureServer(server) {
      server.watcher.add("migrations/");
      server.watcher.add("src/data/**/*.sql");

      server.watcher.on("change", async (file) => {
        if (file.includes("migrations/")) {
          const dbTypes = await parseMigrations("migrations/");
          await generateDbTypes(dbTypes, "src/data/db-types.d.ts", tableNameOverrides);
        }
        if (file.endsWith(".sql")) {
          const dir = path.dirname(file);
          const files = await glob(`${dir}/*.sql`);
          await generateQueryBarrel(dir, files, tableNameOverrides);
        }
      });
    },
  };
}
```

## 4. Generated Output

For each directory containing `.sql` files, the plugin generates two files:

### `queries.generated.ts` — The typed query map

```ts
// src/data/tenet/queries/queries.generated.ts
// AUTO-GENERATED — do not edit

import findBySlug from "./findBySlug.sql";
import getOptions from "./getOptions.sql";
import getWithProposer from "./getWithProposer.sql";
import insertOption from "./insertOption.sql";
import listWithProposer from "./listWithProposer.sql";
import updateStatus from "./updateStatus.sql";

export const queries = {
  findBySlug,
  getOptions,
  getWithProposer,
  insertOption,
  listWithProposer,
  updateStatus,
} as const;
```

### `queries.generated.d.ts` — The type map

```ts
// src/data/tenet/queries/queries.generated.d.ts
// AUTO-GENERATED — do not edit

import type { Tenet, TenetOption } from "../../db-types";
import type { TenetStatus } from "../model";

export declare const queries: {
  readonly findBySlug: string;
  readonly getOptions: string;
  readonly getWithProposer: string;
  readonly insertOption: string;
  readonly listWithProposer: string;
  readonly updateStatus: string;
};

export interface QueryMap {
  findBySlug: {
    params: { slug: string };
    result: Tenet;
  };
  getOptions: {
    params: { tenetId: number };
    result: TenetOption;
  };
  getWithProposer: {
    params: { slug: string };
    result: Tenet & { proposer_login: string; proposer_avatar: string | null };
  };
  insertOption: {
    params: {
      tenetId: number;
      title: string;
      description: string | null;
      pros: string | null;
      cons: string | null;
      sortOrder: number;
    };
    result: void;
  };
  listWithProposer: {
    params: {};
    result: Tenet & { proposer_login: string; proposer_avatar: string | null };
  };
  updateStatus: {
    params: { id: number; status: TenetStatus };
    result: void;
  };
}
```

### `.sql.d.ts` — Per-file module declarations

Each `.sql` file also gets a `.d.ts` so direct imports work:

```ts
// findBySlug.sql.d.ts
declare const sql: string;
export default sql;
```

## 5. Repository Integration

### Enhanced `RepositoryBase` with typed query helpers

```ts
// src/infrastructure/RepositoryBase.ts (additions)

export abstract class RepositoryBase<T extends { id: number }> {
  // ... existing methods ...

  /**
   * Type-safe query helpers. Accept a QueryMap, query name, and params.
   * Result and param types are fully inferred from the QueryMap.
   */
  protected typedOne<QM, K extends keyof QM>(
    db: D1Database,
    queries: Record<string, string>,
    name: K,
    ...args: QM[K] extends { params: infer P }
      ? keyof P extends never ? [] : [P]
      : []
  ): Promise<
    (QM[K] extends { result: infer R } ? R : unknown) | null
  > {
    const sql = queries[name as string];
    const params = args[0] as Record<string, unknown> | undefined;
    return params
      ? this.queryOne(db, sql, params)
      : this.queryOne(db, sql);
  }

  protected typedAll<QM, K extends keyof QM>(
    db: D1Database,
    queries: Record<string, string>,
    name: K,
    ...args: QM[K] extends { params: infer P }
      ? keyof P extends never ? [] : [P]
      : []
  ): Promise<(QM[K] extends { result: infer R } ? R : unknown)[]> {
    const sql = queries[name as string];
    const params = args[0] as Record<string, unknown> | undefined;
    return params
      ? this.queryAll(db, sql, params)
      : this.queryAll(db, sql);
  }

  protected typedExec<QM, K extends keyof QM>(
    db: D1Database,
    queries: Record<string, string>,
    name: K,
    ...args: QM[K] extends { params: infer P }
      ? keyof P extends never ? [] : [P]
      : []
  ): Promise<D1Result> {
    const sql = queries[name as string];
    const params = args[0] as Record<string, unknown> | undefined;
    return params
      ? this.execute(db, sql, params)
      : this.execute(db, sql);
  }
}
```

### Refactored `TenetsRepository`

```ts
// src/data/tenet/repo.ts

import { RepositoryBase } from "infrastructure/RepositoryBase";
import { queries, type QueryMap } from "./queries/queries.generated";
import type { Tenet } from "data/db-types";

type Q = QueryMap;

export class TenetsRepository extends RepositoryBase<Tenet> {
  override readonly tableName = "tenets";

  async findBySlug(db: D1Database, slug: string) {
    return this.typedOne<Q>(db, queries, "findBySlug", { slug });
    // Return type: Tenet | null — fully inferred
    // Params: { slug: string } — compile-time validated
  }

  async getOptions(db: D1Database, tenetId: number) {
    return this.typedAll<Q>(db, queries, "getOptions", { tenetId });
    // Return type: TenetOption[] — inferred
  }

  async listWithProposer(db: D1Database) {
    return this.typedAll<Q>(db, queries, "listWithProposer");
    // Return type: (Tenet & { proposer_login: string; proposer_avatar: string | null })[]
    // No params needed — query has none
  }

  async getWithProposer(db: D1Database, slug: string) {
    return this.typedOne<Q>(db, queries, "getWithProposer", { slug });
  }

  async createWithOptions(
    db: D1Database,
    tenet: {
      title: string;
      slug: string;
      context: string;
      proposed_by_id: number;
    },
    options: {
      title: string;
      description?: string;
      pros?: string;
      cons?: string;
    }[],
  ) {
    const row = await this.create(db, tenet as Partial<Tenet>);

    for (let i = 0; i < options.length; i++) {
      const opt = options[i];
      await this.typedExec<Q>(db, queries, "insertOption", {
        tenetId: row.id,
        title: opt.title,
        description: opt.description ?? null,
        pros: opt.pros ?? null,
        cons: opt.cons ?? null,
        sortOrder: i,
      });
    }

    return row;
  }

  async updateStatus(db: D1Database, id: number, status: TenetStatus) {
    await this.typedExec<Q>(db, queries, "updateStatus", { id, status });
  }
}

export const tenetsRepo = new TenetsRepository();
```

### Before vs After

| Before | After |
|---|---|
| `queryOne<TenetRow>(db, queries.findBySlug, { slug })` | `typedOne<Q>(db, queries, "findBySlug", { slug })` |
| Result type: manually specified generic | Result type: inferred from YAML front matter |
| Params: no validation, could pass wrong keys | Params: compile-time validated against front matter |
| Model types: hand-written, can drift from schema | Model types: generated from migrations, always in sync |
| `"findBySlug"` string: no autocomplete | `"findBySlug"` key: autocomplete from `QueryMap` |

## 6. Implementation Milestones

### Phase 1: Schema Codegen (Low effort, immediate value)

1. Write `parse-migrations.ts` — regex-based parser for `CREATE TABLE` statements
2. Write `generate-db-types.ts` — emits `db-types.d.ts` from parsed schema
3. Wire into `buildStart` hook in the Vite plugin
4. Run once, compare output with hand-written models
5. Replace `RepositoryBase<TenetRow>` with `RepositoryBase<Tenet>` etc.

### Phase 2: Front Matter Parsing + SQL Transform (Medium effort)

1. Install `gray-matter` as a dev dependency: `npm install -D gray-matter`
2. Write `parse-front-matter.ts` — thin wrapper around `gray-matter` that extracts `params` and `result` from the YAML data, returning `{ sql: string, params?: Record<string, string>, result?: string }`
3. Add `transform` hook to strip front matter from `.sql` imports using `gray-matter`
4. Verify SQL files still load correctly via `import.meta.glob`

### Phase 3: Query Barrel Generation (Medium effort)

1. Write `generate-query-barrel.ts` — scans `.sql` files in a directory, emits `queries.generated.ts` and `queries.generated.d.ts`
2. Use `gray-matter` to parse YAML front matter from each `.sql` file and build the `QueryMap` interface
3. Map bare table names in `result:` to imports from `db-types.d.ts`
4. Wire into `buildStart` and `configureServer` for dev HMR

### Phase 4: Typed Repository Helpers (Low effort)

1. Add `typedOne`, `typedAll`, `typedExec` to `RepositoryBase`
2. Migrate one repository (e.g., `TenetsRepository`) to validate the DX
3. Migrate remaining repositories
4. Remove `loadQueries` and `QueryLoader.ts` (no longer needed)

### Phase 5: Polish

1. Add validation: warn if a `result:` references a table name that doesn't exist in `db-types.d.ts`
2. Add validation: warn if `@paramName` in SQL doesn't match a key in `params:`
3. Add `.gitignore` entries for `*.generated.ts` and `*.generated.d.ts`
4. Add `db-types.d.ts` to `.gitignore` (regenerated on every build)

## 7. Key Trade-offs

| Trade-off | Impact |
|---|---|
| Generated files in `src/` (noise) | Mitigated by `.gitignore` + `*.generated.*` naming convention |
| YAML type expressions are strings (no validation at write time) | Mitigated by Phase 5 validation warnings |
| CHECK constraint parsing is regex-based (fragile for complex SQL) | Mitigated by falling back to `string` type when parsing fails |
| `typedOne<Q>` requires passing `Q` generic per call | Could be eliminated with a per-repo subclass that binds `Q` at the class level |
| Front matter is a new convention for the team | Low risk — it's additive, old `.sql` files without front matter still work |

## 8. When to Choose Differently

- **If you want full SQL autocomplete in `.sql` files** (not just TypeScript types), you'd need a VS Code extension or language server — that's a separate project beyond a Vite plugin
- **If the YAML front matter feels too disconnected from the SQL**, consider Kysely's `sql<T>\`...\`` tagged templates instead — types live in the same expression as the SQL, but you lose the separate `.sql` file workflow
- **If you want to validate params against the actual SQL** (not just the YAML declaration), you'd need a SQL parser to extract `@paramName` placeholders and cross-reference — feasible but adds complexity to Phase 5
